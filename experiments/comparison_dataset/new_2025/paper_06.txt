Введение. Постоянное увеличение размеров и сложности систем
способствует возрастанию сложности конфигурирования таких систем ручным
способом, поскольку такие системы могут иметь большое количество
параметров, принимающих множество различных значений. К тому же, не
всегда до конца может быть ясна связь между конкретной метрикой и
параметрами, которые на нее влияют. Подобная проблематика наиболее
актуальна для систем, обрабатывающих большие объемы данных.
Автоматическое конфигурирование может стать решением возникающих
проблем подбора оптимальных значений параметров.
Архитектура решения и описание основных компонентов. Сперва
стоит описать область применения решения. Предполагается, что целевая
система может работать как изолированно, так и взаимодействуя с другими
системами. При этом, сбои в работе одного узла могут провоцировать сбои
связанных узлов, что необходимо учитывать на этапе сбора данных.
Также предполагается, что информация о работоспособности системы
(метрики) фиксируется и что к ним можно получить доступ путем отправки
соответствующего HTTP – запроса. Решениями, накапливающими такие
данные, могут быть, например, базы данных временных рядов или другие
решения, с которыми возможно взаимодействовать, отправляя HTTP – запросы.

Администратор взаимодействует исключительно с модулем Manager,
который предоставляет функционал для выполнения CRUD – операций над
основными сущностями системы (они хранятся в TuningDB), а также по
запросу администратора формирует задания на сбор данных или
конфигурирование. Задания передаются через брокера сообщений Apache Kafka
для повышения надежности – таким образом, сохраняется история заданий и
при отказе одного из основных компонентов она не будет утеряна.
Модуль Collector выполняет функции сбора мониторинговых данных.
Приложение по умолчанию поддерживает два источника данных – InfluxDB и
Prometheus, однако также есть возможность использовать внешние API. Собрав
необходимые данные, модуль формирует csv – файл с записями в формате
“timestamp, value, metric_name”, сохраняет его в хранилище S3 Minio и
отправляет в Apache Kafka сообщение со статусом выполнения задания.
Tuning Module занимается конфигурированием. Брокер сообщений
поставляет ему сформированные менеджером задания, содержащие, помимо
всего прочего, состав и историю изменения значений параметров, а также
имена файлов в S3 Minio, в которых находятся необходимые мониторинговые
данные. Далее модуль с использованием алгоритмов машинного обучения
подбирает оптимальную конфигурацию целевой системы.

Процесс анализа метрик и подбора оптимальной конфигурации
можно разбить на несколько этапов:
1. Предобработка. Необходимо обработать анализируемые
мониторинговые данные и данные об истории изменений параметров и
сформировать записи вида “timestamp, param1_value,..,paramN_value,
metric_delta, workload_type”, где timestamp – временная метка момента
применения конфигурации, paramN_value – значение конфигурационного
параметра целевой системы в текущей конфигурации, metric_delta – разница
между скользящими средними значений метрики в предыдущей конфигурации
322
и текущей, workload_type – тип рабочей нагрузки (например, high/avg/low),
пороги для вычисления которого могут задаваться администратором;
2. Выбор наиболее важных параметров. Для каждого типа рабочей
нагрузки необходимо выявить набор параметров, которые наиболее сильно
влияют на метрику, чтобы сэкономить вычислительные ресурсы и временные
затраты на обучение модели. В этих целях можно использовать, например,
метод SHAP [0];
3. Обучение модели. Для каждого типа рабочей нагрузки необходимо
обучить модель, опираясь на данные из первого шага.
4. Генерация возможных конфигураций. Формируем набор
возможных конфигураций;
5. Нахождение оптимальной конфигурации. Для каждого типа
рабочей нагрузки с помощью соответствующей модели предсказываем
значение метрики при каждой конфигурации и выявляем оптимальную.

типа рабочей нагрузки.
Заключение. За счет использования мониторинговых систем и
гибкой фильтрации входных данных на основе аварийных оповещений и схемы
воздействия, систему можно применить к широкому спектру ПО. Также,
вследствие реализации REST-API подхода, будет упрощена интеграция
системы с внешними сервисами, что можно использовать для автоматизации
различных процессов. Использование методов машинного обучения позволяет
выявлять зависимость метрик от параметров конфигурации и использовать эти
зависимости для подбора оптимальных конфигураций.
